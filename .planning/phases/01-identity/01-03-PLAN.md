<task type="auto">
  <name>Task 1: Build signup.html + js/signup.js and login.html + js/login.js</name>
  <files>signup.html, js/signup.js, login.html, js/login.js</files>
  <action>
    Build two page pairs. Each HTML file links css/app.css and loads its JS module via `<script type="module">`. Each JS file imports from the shared modules in Plan 02.

    **signup.html structure:**
    - `<head>`: viewport meta, Google Fonts (Space Grotesk + Righteous), link to /css/app.css, title "Join Mixmomnt"
    - `<body class="page-loading">`: navbar with .nav-logo linking to /, centered .form-card with:
      - Heading "Create your account" (Righteous font)
      - Subtext "No face required."
      - `<form id="signup-form">`: email input, password input (min 8 chars), confirm-password input
      - Submit button "Create account" (.btn-primary, full width)
      - Link to /login.html: "Already have an account? Log in"
    - `<script type="module" src="/js/signup.js">`

    **js/signup.js logic:**
    1. On DOMContentLoaded: call redirectIfAuth() — if already logged in, redirect away
    2. Remove page-loading class from body after auth check
    3. On form submit: prevent default, disable submit button (set disabled + text "Creating..."), clear errors
    4. Validate: email not empty, password >= 8 chars, passwords match — show inline .form-error if not
    5. Call supabase.auth.signUp({ email, password, options: { emailRedirectTo: 'https://mixmomnt.com/login.html' } })
    6. On error: show error.message below form, re-enable button
    7. On success with data.session (email confirmation disabled): redirect to /profile-setup.html
    8. On success with data.session === null (email confirmation enabled): show message "Check your inbox to confirm your email, then log in." Hide form, show message. Do NOT redirect.

    **login.html structure:**
    - Same `<head>` pattern, title "Log in to Mixmomnt"
    - `<body class="page-loading">`: navbar, centered .form-card with:
      - Heading "Welcome back"
      - `<form id="login-form">`: email input, password input
      - Forgot password link (/forgot-password.html) — right-aligned below password field
      - Submit button "Log in" (.btn-primary, full width)
      - Link to /signup.html: "New here? Create an account"
    - `<script type="module" src="/js/login.js">`

    **js/login.js logic:**
    1. On DOMContentLoaded: call redirectIfAuth() — if logged in with complete profile, skip to feed
    2. Remove page-loading class after check
    3. On form submit: disable button ("Logging in..."), clear errors
    4. Validate: email not empty, password not empty
    5. Call supabase.auth.signInWithPassword({ email, password })
    6. On error: show error.message, re-enable button
    7. On success: check profile completeness:
       - `supabase.from('profiles').select('display_name').eq('id', data.session.user.id).single()`
       - If display_name is null/empty: redirect to /profile-setup.html
       - If display_name exists: redirect to /feed.html (will be built in Phase 3; page may not exist yet — redirect anyway)

    Also add a logout handler that other pages can trigger: export a `logout()` function in js/login.js that calls `supabase.auth.signOut()` then redirects to /login.html. (AUTH-03 — logout can be called from any page's nav.)

    Actually: put logout in a dedicated export in js/auth-guard.js instead of login.js, so any page can import it without importing login logic. Add to auth-guard.js:

    ```javascript
    export async function logout() {
      await supabase.auth.signOut()
      window.location.href = '/login.html'
    }
    ```

    Then any page's nav "Log out" button calls `import { logout } from '/js/auth-guard.js'` inline or via its own page script.

    UX requirements for both pages (from ui-ux-pro-max skill):
    - All inputs: min 44px height (touch target)
    - Labels: always present with `for` attribute matching input `id`
    - Focus states: visible amber ring (already in css/app.css)
    - Error messages: appear immediately below the relevant field, role="alert" for screen readers
    - No layout shift on error appearance — reserve space with min-height on error containers
    - prefers-reduced-motion respected via css/app.css (no action needed in JS)
  </action>

  <name>Task 2: Build forgot-password.html + js/forgot-password.js and update-password.html + js/update-password.js</name>
  <files>forgot-password.html, js/forgot-password.js, update-password.html, js/update-password.js</files>
  <action>
    Build the two-page password reset flow (AUTH-04).

    **forgot-password.html structure:**
    - `<head>`: same pattern, title "Reset your password — Mixmomnt"
    - `<body class="page-loading">`: navbar, centered .form-card with:
      - Heading "Forgot your password?"
      - Subtext: "Enter your email and we'll send you a reset link."
      - `<form id="forgot-form">`: email input
      - Submit button "Send reset link" (.btn-primary, full width)
      - Link back to /login.html: "Back to log in"
    - `<script type="module" src="/js/forgot-password.js">`

    **js/forgot-password.js logic:**
    1. On DOMContentLoaded: remove page-loading class (no auth check needed — this page is public)
    2. On form submit: disable button ("Sending..."), clear errors
    3. Validate: email not empty
    4. Call: `supabase.auth.resetPasswordForEmail(email, { redirectTo: 'https://mixmomnt.com/update-password.html' })`
    5. On error: show error.message, re-enable button
    6. On success: hide form, show success message: "Check your inbox. If this email is registered, you'll receive a reset link shortly." (Do not confirm whether email exists — security best practice)
    7. Re-enable button after 60 seconds to allow retry (prevents spam clicking but allows genuine retry)

    **update-password.html structure:**
    - `<head>`: same pattern, title "Set new password — Mixmomnt"
    - `<body class="page-loading">`: navbar, centered .form-card with:
      - Heading "Set a new password"
      - `<form id="update-form">`: new-password input, confirm-password input
      - Submit button "Update password" (.btn-primary, full width)
      - A "waiting..." loading state shown by default until the PASSWORD_RECOVERY event fires
    - `<script type="module" src="/js/update-password.js">`

    **js/update-password.js logic:**
    1. On load: show a loading/waiting state (e.g., "Verifying your reset link...") — the page cannot do anything until Supabase fires the PASSWORD_RECOVERY auth event via the URL token in the redirect link
    2. Register: `supabase.auth.onAuthStateChange(async (event, session) => { ... })`
    3. On event === 'PASSWORD_RECOVERY':
       - Hide the loading state, show the new password form
       - Remove page-loading class from body
    4. On form submit: disable button ("Updating..."), clear errors
    5. Validate: password >= 8 chars, passwords match
    6. Call: `supabase.auth.updateUser({ password: newPassword })`
    7. On error: show error.message, re-enable button
    8. On success: show "Password updated!" message briefly, then redirect to /login.html after 2 seconds
    9. On event === 'SIGNED_IN' without prior PASSWORD_RECOVERY (edge case): redirect to /profile-setup.html

    Critical: The PASSWORD_RECOVERY event fires because Supabase appended a session token to the redirect URL. The page must NOT try to manually parse URL params — the Supabase client handles this automatically via the hash fragment. Just listen for the event.

    UX: If user navigates to update-password.html without a valid reset link (no token), the PASSWORD_RECOVERY event never fires. After 10 seconds on the loading state, show: "This link has expired or is invalid. Request a new one." with a link to /forgot-password.html.
